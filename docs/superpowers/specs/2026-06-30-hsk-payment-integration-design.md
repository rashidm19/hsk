# HSK Prep — Payment integration (cross-domain via pay.studybox.kz)

- **Date:** 2026-06-30
- **Status:** Approved design, hardened by completeness review (round 2) — ready for implementation plan
- **Scope of this repo:** HSK Prep (`hskprep.cc`) static site + its Supabase (incl. the new Edge
  Function source). The `pay.studybox.kz` backend is a separate system (StudyBox IELTS infra),
  built by the founder, and is specified here only as an interface contract.

## Problem

HSK Prep needs to take real money. The MVP must reuse the **existing acquiring** already running in
the sibling product *StudyBox IELTS tests*, to avoid new merchant paperwork. The acquiring is bound
by the provider's rules to the `studybox.kz` domain, so it cannot be embedded directly on
`hskprep.cc`.

Today payment is **simulated**: `startCheckout()` in `onboarding.js` calls `simulatePayment()`,
which writes `localStorage.hsk_subscription` + the Supabase profile and jumps to the success screen.
This is the single seam to replace.

## Constraints

- HSK Prep is a **static** site (no app server) — plain HTML/CSS/JS pre-rendered by `build.js`,
  deployed on DigitalOcean App Platform. It cannot hold a signing secret or verify a payment itself.
- The funnel lives at `/quiz/`, generated by `buildQuizFunnel()` from `onboarding.js` +
  `data/onboarding.json`. Edit the **sources**, then `node build.js` — never the generated
  `quiz/index.html`. The funnel is `body.lp`, so `scripts/inject-auth.js` does not touch it.
- By the time the user reaches checkout they already have a Supabase session: the email gate (S17)
  is **hard, no skip**. So `user.id` + `email` are available at the pay button.
- Content is currently **not** gated by subscription (`auth-guard.js` checks only for a session).
  The whole site is a static pre-render for SEO; hard-gating it would kill indexation.

## Decisions (locked)

1. **Trust model — server writes entitlement.** Access is granted from a server-verified record,
   not from the return redirect. A static client cannot trust a URL parameter.
2. **Source of truth — existing HSK Supabase**, column `profiles.subscription` (jsonb).
3. **Write mechanism — Supabase Edge Function + HMAC.** The StudyBox backend calls a
   `grant-entitlement` Edge Function on the HSK Supabase project with an HMAC-signed body; the
   function verifies the signature and writes with the service-role key *inside* Supabase. The
   service-role key never leaves Supabase.
4. **Scope — record payment only.** Real payment flows, subscription is written and read, the funnel
   completes and hands off to the platform. **Content gating stays deferred** — SEO untouched.
5. **Currency — KZT.** Prices: `1mo` = 39 000 ₸, `3mo` = 54 000 ₸, `12mo` = 149 000 ₸
   (charged = displayed).
6. **Billing model — one-time, fixed-term.** A single charge buys access for 1 / 3 / 12 months,
   **no auto-renewal, no card-on-file.** No mandate/token storage, no rebill, no in-app cancel
   surface. The funnel copy must be reworded to match (it currently promises auto-renewal).
7. **Analytics — deferred to a fast-follow.** This task leaves named event seams in code
   (`begin_checkout`, `purchase`, `payment_cancelled`) but wires **no** GA4 / Yandex Metrika
   counters. Conversion tracking + ad attribution ship as the next task.
8. **Cross-repo split** — this repo ships the HSK side **and the Edge Function source**; the
   `pay.studybox.kz` backend is built separately by the founder.

## End-to-end flow

1. User clicks **Pay** on `hskprep.cc/quiz/` (static).
2. HSK redirects to `pay.studybox.kz/checkout` with `product`, `plan` (id only — never an amount),
   `uid`, `email`, `return`, `cancel`.
3. StudyBox backend maps `plan → authoritative KZT amount`, creates a **one-time** acquiring order
   with metadata `{ product:'hsk', hsk_uid, plan, months }`, shows the acquiring form on `studybox.kz`.
4. User pays; the acquiring confirms.
5. **Two parallel outcomes:**
   - **5a (server, source of truth):** acquiring → StudyBox webhook (provider-signed) → backend
     POSTs the HSK `grant-entitlement` Edge Function (HMAC) → `profiles.subscription` set to `active`.
   - **5b (UX only):** user redirected back to `www.hskprep.cc/quiz/?pay=success`.
6. HSK reads `profiles.subscription`, polling briefly until `active`, shows the success screen (S25),
   hands off to `/exams/`. The redirect (5b) alone grants nothing; only the verified webhook (5a) can.

## Components

### A. HSK Prep side (this repo)

**A.1 — Replace the payment seam** (`startCheckout(tier, discount)`, ~`onboarding.js:861`):
- Read `user.id` + `email` from the Supabase session (`HSKAuth.getUser()`).
- Build return/cancel URLs on `www.hskprep.cc`: `/quiz/?pay=success`, `/quiz/?pay=cancel`.
- `location.href = https://pay.studybox.kz/checkout?product=hsk&plan=<tier.id>&uid=<user.id>&email=<email>&return=<enc>&cancel=<enc>`.
- **`plan` id only — never an amount.** `uid` is the sole grant key; `email` is display/receipt only.
- Tear down the S23 modal focus-trap + countdown timer before navigating.
- Fire a `begin_checkout` **seam** (no-op stub now; analytics wired in the fast-follow).
- Unconfigured-Supabase preview keeps the simulated path so static preview still runs.

**A.2 — Return handling** (`init()`, ~`onboarding.js:909`) — exact ordering matters:
1. Read `?pay` **before** the `LS_DONE` early-return guard (`onboarding.js:919`).
2. `?pay=success` → force a polling/S25 screen regardless of restored `state.idx`; poll
   `getSubscription(uid)` ~6 × 1 s for `status:'active'`. Set `LS_DONE` **only** after the poll
   confirms `active` — never at checkout, never on the timeout branch.
3. After the timeout (webhook lagging): since content is ungated, proceed to S25/handoff
   optimistically; the webhook-written row applies on the next authenticated visit.
4. **Null session on return** (mobile in-app webviews / system-browser return can drop the session):
   guard the poll so a null `user` never throws; show S25/handoff optimistically.
5. `?pay=cancel` → `goById('s22')` then programmatically `openExitIntent()`; grant nothing.
   (`A.plan` / `A.discount` already persist in `localStorage`, so no new persistence is needed.)
   Requires hoisting/exposing `openCheckout`/`openExitIntent` (currently inner fns ~`onboarding.js:788/820`).
6. Keep the `LS_DONE` guard **after** the `?pay` branch so a fresh visit still fast-forwards.

**A.3 — New read path.** Add `HSKAuth.getSubscription(userId)` →
`from('profiles').select('subscription').eq('id',userId).maybeSingle()` returning `data.subscription`
(`profiles_select_own` RLS already authorizes the owner — no policy change). Define distinct return
values for *not-yet-active*, *no-session*, and *error* so the poll can tell them apart.

**A.4 — Remove all client-side subscription writes** (B2):
- Drop `'subscription'` from the `updateProfile` allow-list (`auth.js:208`); the client persists only
  `onboarding` answers.
- Remove the `subscription` write from `syncToProfile()` (`onboarding.js:898`) and from
  `markPurchased()` for the real path. (Belt-and-suspenders DB trigger in §D.)

**A.5 — Currency-aware `fmtPrice`** (B4): format from `pricing.currency`. KZT → integer tenge,
space-grouped thousands, `₸` suffix (`39 000 ₸`), no decimals. Keep a USD branch for local preview.
Set `pricing.currency = "KZT"` and tier `price`/`base`/`perDay` to whole tenge
(39000 / 54000 / 149000; `perDay` ≈ 1300 / 600 / 408). After `node build.js`, verify the tier cards,
`base` strikethrough, "Total due today", `Subscribe · {price}` CTA, and the disclosure all render
`… ₸`.

**A.6 — One-time copy** (B1, `data/onboarding.json`):
- `s23.disclosure` (line 270) → one-time terms, e.g. *"You'll be charged {price} once for {plan} of
  access. This is a one-time payment — no auto-renewal."* Drop "every {interval} until you cancel"
  and "Cancel anytime in your account settings". Keep Terms / Privacy / Refund references.
- `s22.riskReversal` (line 253) "No commitment. Cancel anytime." → e.g. *"One-time payment. No
  auto-renewal."*
- `s23.paymentNote` (line 266) → reword for a **full-page redirect** (*"You'll be taken to our
  secure payment page to enter your card."*) and drop "Final amount may vary due to your bank's
  exchange rate" (KZT, no FX for local cards).
- The separate **Pass guarantee** refund (`guarantee.*`, s24) stays — it is not auto-renew.

**A.7 — Rebuild.** `node build.js` regenerates `quiz/index.html`.

### B. StudyBox backend side (separate system — contract only)

1. `GET /checkout` — validate `product=hsk`, map `plan → KZT amount` + `months`, create a **one-time**
   acquiring order with metadata `{ product:'hsk', hsk_uid, plan, months }`, render the form on
   `studybox.kz`.
2. **Acquiring webhook** — verify the provider's signature, idempotent by `order_id`, on success POST
   the HSK `grant-entitlement` Edge Function with an HMAC-signed body (incl. computed `expires_at`).
3. **Return redirect** — `return` URL on success, `cancel` URL otherwise (optionally
   `&reason=declined|abandoned`).
4. **Secrets** — holds the shared HMAC secret + the Edge Function URL (not the Supabase service-role key).

### C. HSK Supabase Edge Function `grant-entitlement`

- **Source committed in this repo** at `supabase/functions/grant-entitlement/index.ts` (Deno).
- **Deploy `verify_jwt=false`** so HMAC is the sole authn (otherwise the JWT-less StudyBox call 401s
  at the gateway). Server-to-server only — emit **no CORS headers**.
- **Secret:** `HSK_GRANT_HMAC_SECRET` via `supabase secrets set` (documented next to `schema.sql`);
  never in `config/auth.js` (ships to browser). Service-role from the auto-injected
  `SUPABASE_SERVICE_ROLE_KEY`.
- **Verify:** constant-time HMAC-SHA256 over the **raw request body**; reject on mismatch.
  Replay defense: signed payload includes `ts` (+ `nonce`); reject if `|now-ts| > ~300s` or if
  `order_id` is already terminal.
- **Determinism:** re-derive `amount` / `interval` / `expires_at` from a server-side plan map keyed
  by the validated `plan` id; store the body's amount in `payments.raw` as audit; reject `currency != 'KZT'`.
- **Write:** insert `payments` first (audit survives even if the entitlement target is missing), then
  **upsert** `profiles` on `id=uid` (`insert … on conflict (id) do update set subscription=…`) so a
  missing profile row doesn't silently no-op; log a warning if 0 rows updated.
- **Logging:** one structured line per invocation (`order_id, uid, plan, amount, hmac_ok, idempotent,
  result, reject_reason`); persist failures durably (they outlive Edge log retention).

### D. Data model (`supabase/schema.sql`)

- **`payments` table** (audit + idempotency): `order_id text primary key`,
  `user_id uuid references auth.users`, `plan text`, `amount numeric`, `currency text`,
  `status text`, `receipt text` (provider ref for support), `paid_at timestamptz`, `raw jsonb`,
  `created_at timestamptz default now()`. **`alter table payments enable row level security;` with
  ZERO policies** — service-role bypasses RLS; the public anon key can read nothing (financial/PII).
- **Lock `profiles.subscription`** (B2): a `before update` trigger that raises unless
  `auth.role() = 'service_role'` is changing the column — RLS can't do per-column `WITH CHECK`.
- **Canonical `subscription` shape** — one schema all writers agree on:
  `{ status:'active', plan, price, currency:'KZT', interval, provider, order_id, paid_at, expires_at }`.
  Make `simulatePayment()` emit the same keys (`provider:'simulated'`, synthetic `order_id`, computed
  `expires_at`). Readers strictly require only `status`; everything else is display/support metadata.
- **`expires_at`** is computed **by the Edge Function** from `paid_at + plan.months` (UTC) using the
  plan map — not trusted from the body. For MVP nothing flips `active → expired` (gating is deferred)
  — QA should not treat a permanently-active row as a bug.
- **Idempotent migration** — append `payments` + any policies guarded with
  `drop policy if exists …; create policy …` (mirroring the file's existing drop-if-exists for the
  trigger) so `schema.sql` stays re-runnable by hand.
- **Refund seam** (no logic now): don't hard-code `'active'` as the only writable status; reserve
  `'refunded'` / `'canceled'` so a future refund webhook can flip `payments.status` + `subscription`.

## Interface contract

- **HSK → pay.studybox.kz** (browser GET): `product, plan, uid, email, return, cancel`. No amount.
  `uid` is the sole grant key; `email` is display/receipt only (mismatch with the card's billing
  identity is expected and harmless).
- **acquiring → StudyBox** (webhook): provider-signed; carries `{ hsk_uid, plan, months, order_id }`.
- **StudyBox → HSK Edge Function** (POST + HMAC): canonical JSON
  `{ uid, plan, order_id, currency:'KZT', paid_at, ts, receipt? }`. The function **re-derives**
  amount / interval / expires_at from the `plan` id via the plan map — any `amount`/`months`/
  `expires_at` in the body are audit-only and are NOT part of the entitlement, so they are kept out
  of the signed contract to avoid drift. Replay is covered by `order_id` idempotency + `ts` freshness
  (±300 s); no separate nonce. Signature: HMAC-SHA256 over the **raw body bytes** (no
  re-serialization), hex-encoded, in header `X-HSK-Signature`. Both sides must agree byte-for-byte.
- **HSK reads:** `profiles.subscription.status === 'active'`.
- **Plan → KZT map** (must match `data/onboarding.json`; `onboarding.json` is canonical, the
  contract table is the authority StudyBox copies): `1mo=39000`, `3mo=54000`, `12mo=149000`.
  Add a launch-checklist diff step; optionally have the function reject a webhook amount that
  doesn't match the expected KZT value (fail closed on drift).

## Security notes

- Only a real, provider-verified payment can trigger the entitlement write — the redirect cannot.
- Amount is authoritative on the backend (`plan → amount`); the client never sends a price.
- Client can no longer write `subscription` (allow-list removed + DB trigger) — closes self-grant.
- Service-role key stays inside Supabase; StudyBox holds only the HMAC secret + function URL.
- HMAC: constant-time compare, `ts`/`nonce` freshness (replay ≠ idempotency), idempotency by
  `order_id`, reject unknown `plan` and non-KZT currency.
- `payments` RLS enabled with zero policies — anon key reads nothing.
- Tampering `uid`/`plan` in the redirect only lets an attacker pay *for someone* (`uid` is the user's
  own non-enumerable UUID) — not an attack.

## Edge cases

- **Webhook/redirect race (5a vs 5b):** poll on return; content ungated → proceed even if the record
  lags; it applies on the next visit.
- **Null session on return** (in-app webviews): guard the poll; show S25/handoff optimistically.
- **Cancel / abandon:** `?pay=cancel` → `goById('s22')` + `openExitIntent()`.
- **Double payment / webhook retries:** idempotent by `order_id`; `ts` freshness blocks stale replay.
- **Missing profiles row:** function upserts; always writes `payments` first.
- **Lost webhook (support fire-drill):** `grant-entitlement` is safely replayable — support re-POSTs a
  known `order_id`; max (service-role) has a documented read/replay query; runbook: read the StudyBox
  order → invoke the function → verify `profiles.subscription` flipped.
- **Unconfigured Supabase (local preview):** keep the simulated path.

## Out of scope

- **Content gating** (closing `body.app` pages behind a subscription) — deferred to preserve SEO.
- **Recurring / auto-renew machinery** — Decision 6 is one-time; no mandate/rebill/cancel.
- **Refund execution** — only a status seam is reserved (§D).
- **Building the StudyBox backend** (component B) — separate system, built by the founder.

## Deferred — fast-follow (next task)

- **Conversion analytics:** fire `purchase` (GA4 + Yandex Metrika) only after the poll confirms
  `active`, value = charged KZT integer, deduped by `order_id`; plus an authoritative server-side
  conversion (GA4 Measurement Protocol + Metrika offline upload) from the function for users who never
  return. Also `begin_checkout` / `payment_cancelled`. Seams are left in place by this task.
- **Ad attribution:** capture `utm_*` / `yclid` / `gclid` at funnel entry, preserve across the
  cross-domain hop, persist click-id in `payments` for offline-conversion upload.

- **Round 2:** billing = one-time fixed-term (no auto-renew); analytics deferred to fast-follow
