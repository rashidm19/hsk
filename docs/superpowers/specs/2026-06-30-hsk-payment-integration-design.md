# HSK Prep — Payment integration (cross-domain via pay.studybox.kz)

- **Date:** 2026-06-30
- **Status:** Approved design — ready for implementation plan
- **Scope of this repo:** HSK Prep (`hskprep.cc`) static site + its Supabase. The
  `pay.studybox.kz` backend is a separate system (StudyBox IELTS infra) and is specified
  here only as an interface contract.

## Problem

HSK Prep needs to take real money. The MVP must reuse the **existing acquiring** already
running in the sibling product *StudyBox IELTS tests*, to avoid new merchant paperwork. The
acquiring is bound by the provider's rules to the `studybox.kz` domain, so it cannot be
embedded directly on `hskprep.cc`.

Today payment is **simulated**: `startCheckout()` in `onboarding.js` calls `simulatePayment()`,
which writes `localStorage.hsk_subscription` + the Supabase profile and jumps to the success
screen. This is the single seam to replace.

## Constraints

- HSK Prep is a **static** site (no app server) — plain HTML/CSS/JS pre-rendered by `build.js`,
  deployed on DigitalOcean App Platform. It cannot hold a signing secret or verify a payment
  server-side on its own.
- The funnel lives at `/quiz/`, generated by `buildQuizFunnel()` from `onboarding.js` +
  `data/onboarding.json`. Edit the **sources**, then `node build.js` — never the generated
  `quiz/index.html`.
- By the time the user reaches checkout they already have a Supabase session: the email gate
  (S17) is **hard, no skip** (`onboarding.js` — "verifying the emailed code is required").
  So `user.id` + `email` are available at the pay button.
- Content is currently **not** gated by subscription (`auth-guard.js` checks only for a session).
  The whole site is a static pre-render for SEO; hard-gating it would kill indexation.

## Decisions (locked)

1. **Trust model — server writes entitlement.** Access is granted from a server-verified record,
   not from the return redirect. A static client cannot trust a URL parameter.
2. **Source of truth — existing HSK Supabase**, column `profiles.subscription` (jsonb). HSK already
   reads it via `HSKAuth.getUser()`; almost no new read code on the HSK side.
3. **Write mechanism — Supabase Edge Function + HMAC.** The StudyBox backend calls a
   `grant-entitlement` Edge Function on the HSK Supabase project with an HMAC-signed body. The
   function verifies the signature and writes with the service-role key *inside* Supabase. The
   service-role key never leaves Supabase (least privilege).
4. **Scope — record payment only.** Real payment flows, subscription is written and read, the
   funnel completes and hands off to the platform. **Content gating stays deferred** (separate
   task) — SEO untouched.
5. **Currency — KZT.** The Kazakhstani acquiring charges in tenge. The displayed funnel price
   must equal the charged price, so tier prices in `data/onboarding.json` move to KZT and must
   match the backend `plan → amount` map.

## End-to-end flow

1. User clicks **Pay** on `hskprep.cc/quiz/` (static).
2. HSK redirects to `pay.studybox.kz/checkout` with `product`, `plan` (id only — never an amount),
   `uid`, `email`, `return`.
3. StudyBox backend maps `plan → authoritative KZT amount`, creates an acquiring order with
   metadata `{ product: 'hsk', hsk_uid }`, and shows the acquiring form on `studybox.kz`.
4. User pays; the acquiring confirms.
5. **Two parallel outcomes:**
   - **5a (server, source of truth):** acquiring → StudyBox webhook (provider-signed,
     server-to-server) → backend calls the HSK `grant-entitlement` Edge Function (HMAC) →
     `profiles.subscription` set to `active` in HSK Supabase.
   - **5b (UX only):** user is redirected back to `hskprep.cc/quiz/?pay=success`.
6. HSK reads `profiles.subscription`, polling briefly until `active`, then shows the success
   screen (S25) and hands off to `/exams/`.

The redirect (5b) on its own grants nothing; only the verified webhook (5a) can.

## Components

### A. HSK Prep side (this repo — the implementation)

1. **Replace the payment seam** — `startCheckout(tier, discount)` in `onboarding.js`
   (currently near line 861). Instead of `simulatePayment()`:
   - read `user.id` + `email` from the Supabase session (`HSKAuth.getUser()`);
   - build the return URL `https://www.hskprep.cc/quiz/?pay=success` (and a cancel URL);
   - `location.href = https://pay.studybox.kz/checkout?product=hsk&plan=<tier.id>&uid=<user.id>&email=<email>&return=<encoded return>&cancel=<encoded cancel>`;
   - **send only `plan` id**, never an amount.
   - Keep a graceful fallback when Supabase is unconfigured (local preview): keep simulating,
     so static preview still works.
2. **Return handling** — in `init()` (currently near line 909):
   - `?pay=success` → poll `profiles.subscription` (~6 × 1s) for `status:'active'`; on success
     mark done, show S25, hand off. If still not active after the window, proceed to the platform
     anyway (content is not gated) and let the record catch up.
   - `?pay=cancel` → reopen the checkout / exit-intent screen; grant nothing.
3. **localStorage is a UX cache only** — the authoritative subscription is always the Supabase row.
4. **Pricing copy** — move tier prices in `data/onboarding.json` (`pricing.tiers[].price`/`base`/
   `perDay`) to **KZT**, matching the backend `plan → amount` map. Keep the disclosure copy
   accurate. (Exact numbers are a business decision — left as fill-in.)
5. **Rebuild** — `node build.js` to regenerate `quiz/index.html`; the funnel is `body.lp` so
   `scripts/inject-auth.js` does not touch it.

### B. StudyBox backend side (separate system — contract only)

1. `GET /checkout` — validate `product=hsk`, map `plan → KZT amount`, create the acquiring order
   with metadata `{ product:'hsk', hsk_uid, plan }`, render the acquiring form on `studybox.kz`.
2. **Acquiring webhook** — verify the provider's signature, ensure idempotency by `order_id`,
   and on success call the HSK `grant-entitlement` Edge Function.
3. **Return redirect** — send the user to the `return` URL on success, the `cancel` URL otherwise.
4. **Secret** — holds the shared HMAC secret for the Edge Function (not the Supabase service-role key).

### C. HSK Supabase Edge Function `grant-entitlement`

- Input (POST, JSON): `{ uid, plan, order_id, amount, currency:'KZT', paid_at, interval }` +
  an HMAC signature header over the raw body using the shared secret.
- Verifies HMAC (constant-time), rejects on mismatch.
- Idempotent upsert into `payments` (unique `order_id`), then sets `profiles.subscription` to
  `active` for `uid` using the service-role client (internal to Supabase).
- Deployed via the Supabase MCP tooling.

### D. Data model (`supabase/schema.sql`)

- Extend with a `payments` table for audit + idempotency:
  - `order_id text primary key`, `user_id uuid references auth.users`, `plan text`,
    `amount numeric`, `currency text`, `status text`, `paid_at timestamptz`,
    `raw jsonb`, `created_at timestamptz default now()`.
  - RLS: no client access (written only by the Edge Function via service-role).
- `profiles.subscription` shape (written by the function, read by HSK):
  `{ status:'active', plan, price, currency:'KZT', interval, provider:'studybox', order_id, paid_at, expires_at }`.

## Interface contract summary

- **HSK → pay.studybox.kz** (browser redirect, GET):
  `product, plan, uid, email, return, cancel`. No amount.
- **acquiring → StudyBox** (webhook): provider-signed; carries order metadata incl. `hsk_uid`, `plan`.
- **StudyBox → HSK Edge Function** (POST + HMAC): `uid, plan, order_id, amount, currency, paid_at, interval`.
- **HSK reads**: `profiles.subscription.status === 'active'`.

## Security notes

- Only a real, provider-verified payment can trigger the entitlement write — the redirect cannot.
- Amount is authoritative on the backend (`plan → amount`); the client never sends a price.
- Tampering `uid`/`plan` in the redirect only lets an attacker pay *for someone*, which is not an
  attack; `uid` is the user's own non-enumerable Supabase UUID.
- Service-role key stays inside Supabase; StudyBox holds only the HMAC secret.
- Edge Function: constant-time HMAC compare, idempotency by `order_id`, reject unknown `plan`.

## Edge cases

- **Webhook/redirect race (5a vs 5b):** poll on return; since content is ungated, proceed even if
  the record lags slightly.
- **Cancel / failure:** `?pay=cancel` → back into the funnel, nothing granted.
- **Double payment / webhook retries:** idempotent by `order_id`.
- **Unconfigured Supabase (local preview):** keep the simulated path so static preview still runs.

## Out of scope

- Hard or soft **content gating** (closing `body.app` pages behind an active subscription) — kept
  deferred to preserve SEO; tracked separately.
- Refunds / subscription cancellation / renewal automation beyond recording the initial purchase.
- Building the StudyBox backend endpoints (separate system; max has SSH access).

## Open items / assumptions

1. **Exact KZT prices** per tier — business decision; must match the backend `plan → amount` map.
   Single-sourcing across two repos is out of scope; document the matching values.
2. **StudyBox backend ownership** — implemented separately by max; this repo ships only the HSK side.
3. **Canonical host for `return`** — use `www.hskprep.cc` (the canonical/primary host) for the
   return URL, and ensure it is in the Supabase redirect allowlist if any auth round-trip occurs.
